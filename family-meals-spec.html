Project Overview

🎯 High-Level Goal

Build a WhatsApp-based family meal planning system where:

  • Family starts with zero recipes
  • Members add recipes interactively via /add command
  • Members search BBC Good Food for inspiration via /inspire command
  • Members propose & vote on meals weekly
  • System generates shopping lists automatically
  • All data persists in local JSON/YAML files

⚙️ Key Constraints

Constraint Value
Storage Local files only (no database)
Scale 4 family members, ~50 recipes (estimated max)
Platform WhatsApp via OpenClaw MCP
Hosting Self-hosted on moltbot server
Development Time 4-5 days (MVP)

✅ Success Criteria

  • ✅ Family can add 5 recipes via chat in <30 minutes
  • ✅ Weekly meal planning workflow works end-to-end
  • ✅ Shopping list generates correctly (deduplicated, grouped by category)
  • ✅ BBC Good Food search returns 3+ usable recipes per query
  • ✅ No data loss on bot restart
  • ✅ All edge cases handled gracefully (invalid input, network errors, etc.)

Implementation Checklist

Phase 1: Core Setup (Day 1)

  • Create directory structure (~/.family-meals/)
  • Initialize package.json with dependencies
  • Create config.yaml template
  • Create family_members.yaml template
  • Set up bot.js main entry point
  • Set up unit test framework (Jest)

Phase 2: Core Services (Day 1-2)

  • Implement fileManager.js (readJson, writeJson, readYaml, writeYaml)
  • Implement validators.js (schema validation)
  • Implement formatters.js (WhatsApp formatting)
  • Write unit tests for services

Phase 3: Command Handlers (Day 2-3)

  • /add command (multi-step conversation)
  • /list command (with filtering)
  • /inspire command (BBC search)
  • /propose command (meal suggestions)
  • /vote command (voting system)
  • /plan command (view weekly plan)
  • /done command (lock and generate shopping list)
  • Write integration tests

Phase 4: BBC Integration (Day 2-3)

  • Implement bbcScraper.js with search & caching
  • Create test cases for scraper
  • Test with 5+ real BBC searches

Phase 5: QA & Testing (Day 4)

  • Run unit test suite (>80% coverage)
  • Run integration tests
  • Manual UAT with test account
  • Test error scenarios
  • Performance testing (50 recipes)

Phase 6: Deployment (Day 4-5)

  • Final code review
  • Documentation (README.md)
  • Deploy to production
  • Monitor logs for errors
  • Collect family feedback

Risk Assessment

Risk Impact Likelihood Mitigation
BBC website structure changes High Medium Graceful fallback to /add, fallback endpoint
File corruption on write High Low Atomic writes, backup on error
Concurrent writes (multiple users) High Medium File locking or JSON merges
User forgets conversation step Medium Medium Timeout cleanup (5 min), /cancel command
Rate limit on BBC scraper Medium Medium Cache, local fallback, limit searches to 1/min
